# Kenari CLI — Design Document

> Status: **Planned** (v0.3)

---

## Overview

Kenari CLI is a lightweight agent written in **Rust** that runs on monitored hosts.
It sends metrics to your Kenari gateway without requiring inbound ports on the monitored host.

```
Monitored Host          Edge (Kenari Gateway)
┌──────────────┐        ┌──────────────────┐
│ kenari-cli   │──HTTPS─▶  /api/agent/push │
│ (Rust agent) │        │  (SvelteKit API) │
└──────────────┘        └──────────────────┘
```

---

## Why Rust?

| Criterion | Rust | Go |
|-----------|------|-----|
| Binary size | ~2MB | ~8MB |
| Memory usage | ~2MB RSS | ~10MB RSS |
| Startup time | <10ms | ~50ms |
| Cross-compile | Excellent | Good |
| Single binary | ✅ | ✅ |
| No runtime deps | ✅ | ✅ |

For an agent that runs 24/7 on potentially resource-constrained servers, Rust wins on efficiency.

```bash
# One-liner install (Linux/macOS)
curl -fsSL https://kenari.dev/install.sh | sh

# Or download binary directly

sudo mv kenari-linux-amd64 /usr/local/bin/kenari
```

## Commands

```bash
# Register this host with a Kenari gateway
kenari register \
  --gateway https://monitor.yourdomain.com \
  --token <agent-token-from-kenari-dashboard> \
  --name "prod-server-1"

# Start the agent (runs in foreground)
kenari agent start

# Install as systemd service
kenari agent install

# Check agent status
kenari status

# Send a one-time metric snapshot
kenari push

# View collected metrics
kenari metrics

# Unregister this host
kenari unregister
```

## Metrics Collected

### System (default, always on)
- CPU usage (per core + total)
- Memory usage (used/total/cached)
- Disk usage (per mount point)
- Network I/O (bytes in/out per interface)
- System uptime
- Load average

### Process (optional)
- Top processes by CPU/memory
- Specific process monitoring by name/PID

### Custom (via plugins)
```toml
# /etc/kenari/config.toml
[[plugins]]
name = "nginx"
type = "exec"
command = "nginx -T | grep active_connections"
interval = 30

[[plugins]]
name = "postgres"
type = "postgres"
dsn = "postgresql://localhost/mydb"
query = "SELECT count(*) FROM pg_stat_activity"
```

## Communication Protocol

### Push mode (default)
Agent polls and pushes metrics to gateway every N seconds:

```
POST /api/agent/push
Authorization: Bearer <agent-token>
Content-Type: application/json

{
  "host_id": "prod-server-1",
  "timestamp": 1713456789,
  "metrics": {
    "cpu_percent": 23.4,
    "memory_used_mb": 1024,
    "memory_total_mb": 4096,
    "disk_used_gb": 45.2,
    "disk_total_gb": 100.0,
    "load_1m": 0.45,
    "load_5m": 0.38,
    "load_15m": 0.31
  }
}
```

### WebSocket mode (v0.4, real-time)
For real-time dashboards, agent maintains a WebSocket connection.

## Configuration File

```toml
# /etc/kenari/config.toml

[agent]
gateway = "https://monitor.yourdomain.com"
token = "your-agent-token"
host_name = "prod-server-1"
push_interval = 30  # seconds

[metrics]
cpu = true
memory = true
disk = true
network = true
processes = false  # disabled by default (privacy)

[tls]
verify = true  # set false for self-signed certs
```

---

## Security

- Agent token is a random 256-bit secret generated by Kenari dashboard
- All communication over HTTPS (TLS 1.2+)
- Agent has no inbound ports — push-only model
- Token can be revoked from Kenari dashboard at any time
- Metrics are stored in Kenari's database, not on the agent

## Build from Source

```bash
git clone https://github.com/sandikodev/kenari
cd kenari/cli
cargo build --release
./target/release/kenari --version
```

### Cross-compilation targets
```bash
# Linux x86_64
cargo build --release --target x86_64-unknown-linux-musl

# Linux ARM64 (Raspberry Pi, AWS Graviton)
cargo build --release --target aarch64-unknown-linux-musl

# macOS
cargo build --release --target x86_64-apple-darwin

# Windows
cargo build --release --target x86_64-pc-windows-msvc
```
